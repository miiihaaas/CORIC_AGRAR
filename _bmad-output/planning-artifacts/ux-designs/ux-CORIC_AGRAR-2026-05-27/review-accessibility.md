# Accessibility Review — Ćorić Agrar UX Spines

## Overall verdict

The spines show above-average accessibility literacy — WCAG 2.1 AA is declared, ARIA usage is named, reduced-motion is covered, focus trap and 44px touch floor are explicit. However, several concrete defects undermine the AA claim: gold-500 as a text accent is unsafe on light surfaces (~2:1), the focus ring (green-400 on green-800) fails the 3:1 non-text UI minimum, jeegee-blue passes only large-text contrast, and several interactive surfaces (HTMX requests, sticky nav state, language switch, search expand, lightbox open) lack documented screen-reader announcements. The spines are strong on structural a11y (skip link, keyboard, reduced motion, alt text policy) but weak on dynamic-content announcement, color-system audit beyond a few sampled pairs, and explicit boundary cases (carousel pause control, accordion icon-only toggle, error association IDs).

## Critical (4)

- **Gold-500 (#e7af12) on light backgrounds fails AA for text** (DESIGN.md §Colors, §Typography h1 spec, EXPERIENCE.md §Accessibility Floor only audits gold-on-green-900). Measured ratios: **gold-500 on white = 1.99:1**, **gold-500 on cream = 1.77:1**. DESIGN.md explicitly says hero h1 "često sa zlatnom bojom" and statistike medaljon broj is "gold-500 color" inside a gold-outlined circle (so the digit sits on whatever surface is behind the circle — likely cream or white in light sections). *Fix:* Restrict gold-500 as text/large-digit color to dark green backgrounds only. On light backgrounds use green-900 for the digit and keep gold only for the outline; or darken gold to ~#a87d00 when used as text on cream/white. Add a "Permitted surfaces" table per accent color to DESIGN.md §Colors.

- **Focus ring (green-400 on green-800) fails 3:1 non-text contrast** (EXPERIENCE.md §State Patterns, §Form input focus). Measured **green-400 on green-800 = 2.41:1**, below the WCAG 2.4.11/1.4.11 floor of 3:1 for focus indicators and UI components. The opacity 0.3/0.6 modifiers make it worse. The same ring is specified on form inputs inside dark-green cards (hero overlay forms, footer newsletter). *Fix:* Use a 2-color focus indicator (white or gold-500 outer ring on dark surfaces, green-800 ring on light surfaces) and drop the opacity multiplier on the visible ring itself. Reach ≥3:1 against *every* adjacent surface. Use `:focus-visible` with a solid ring, not a translucent shadow.

- **No live-region pattern documented for HTMX dynamic updates** (EXPERIENCE.md §Forme i HTMX feedback, §Filter, §Search). Filters update results "in-place", success cards replace forms, search dropdown populates as user types, server errors return partial updates — none of these declare `aria-live` regions or how screen readers learn the result. SR users on AT will hear silence on filter change, on form submit success, and on validation errors. *Fix:* Add to §Accessibility Floor: "All HTMX-swapped regions that change without page navigation MUST be wrapped in or announce through an `aria-live="polite"` region. Form errors use `aria-live="assertive"` + `role="alert"`. Filter result count change announces 'N rezultata pronađeno'. Search dropdown uses `role="listbox"` with `aria-activedescendant`."

- **Slider auto-advance fails WCAG 2.2.2 (Pause, Stop, Hide)** (EXPERIENCE.md §Slider Testimonijali, §Animacije). Auto-advance 6s is listed; spec says "hover pauses (desktop)" but provides no **explicit, keyboard-reachable pause/stop control** for keyboard or touch users, no documented pause-on-focus. WCAG 2.2.2 requires a way to pause any auto-moving content lasting >5s. Also missing: respect `prefers-reduced-motion` to disable auto-advance entirely (reduced motion section lists "slide" but ambiguously). *Fix:* Add visible Pause/Play button to the carousel, keyboard-focusable, with `aria-pressed`. Auto-advance MUST stop on focus-within and on `prefers-reduced-motion: reduce`. Document this in §Slider explicitly.

## High (8)

- **Accordion `+`/`−` toggle is icon-only with no `aria-label`** (DESIGN.md §Accordion, EXPERIENCE.md §Accordion). Spec mandates `aria-expanded` and `aria-controls` (good) but the header pattern is "h4 + plus/minus ikona desno". If the visible label is the spec section name ("Motor", "Transmisija") that suffices, but the icon needs to be marked `aria-hidden="true"` so AT doesn't read "plus" or "minus". Also: the rule "ikona se ne rotira — `+` ↔ `−`" means there's no animation-state cue beyond text replacement; works for sighted but ensure `aria-expanded` flips. *Fix:* Specify in EXPERIENCE.md §Accordion: button uses section title as accessible name; icon `aria-hidden="true"`; `aria-expanded` flips on toggle; `aria-controls` points to content panel `id`. Also clarify whether headers are wrapped in `<h3>`/`<h4>` (heading semantics) AND `<button>` (per ARIA APG disclosure pattern).

- **Sticky nav shrink not announced; can hide first focusable items** (EXPERIENCE.md §Sticky nav). On scroll-down nav shrinks and "top header sakriven"; on scroll-up it expands. Keyboard users tabbing into the page from the address bar will hit whichever state happens to be active. If a sighted-keyboard user shrinks by scrolling then tabs backward, focus can land on visually-hidden phone/social links (still in DOM) unless removed. Also: shrunk-state nav at ~60px must keep all interactive targets ≥44px tall (currently a contradiction with 44px floor). *Fix:* Either (a) remove top-header items from tab order in shrunk state via `inert` or `tabindex="-1"`, or (b) keep them visible. Document min target height in shrunk state (44px logo+links area). Add `aria-label` to nav landmark that doesn't change with state.

- **Search expand needs ARIA combobox pattern** (EXPERIENCE.md §Search). "Klik na search ikonu → ekspanduje u inline tekstualno polje" — input appears dynamically with autocomplete dropdown grouped by type. Spec is silent on `role="combobox"`, `aria-expanded`, `aria-controls`, `aria-activedescendant`, and announcement of result count. Also: "klik van zatvori search" is the only close path documented — keyboard users have Esc returning focus to the icon (good) but tab-out behavior is undefined. *Fix:* Use ARIA 1.2 combobox pattern. Add: "On expand, focus moves to input. Tab out of input commits selection or closes. Result count announced via `aria-live`. Grouped results use `role="listbox"` with `role="group"` per type."

- **Lightbox spec missing: `aria-labelledby`, image alt within dialog, prev/next button labels** (EXPERIENCE.md §Lightbox). `role="dialog"`, `aria-modal`, focus trap, Esc — all good. Missing: dialog needs accessible name (`aria-labelledby` pointing to image title/index "Slika 2 od 8"); prev/next buttons need `aria-label="Prethodna slika" / "Sledeća slika"`; image inside dialog must keep meaningful alt (not `alt=""`). Swipe is keyboard-inaccessible — arrow keys are mentioned, good. *Fix:* Add to §Lightbox: dialog accessible name pattern, button labels, image counter announcement on prev/next, focus return target verification.

- **Language switcher does not set `lang` on `<html>` programmatically or announce locale change** (EXPERIENCE.md §Lokalizacija switcher). Spec describes the URL redirect but not the `<html lang="sr|hu|en">` attribute or how AT handles a full-page locale change. Also, the fallback `ⓘ` icon explanation is delivered via tooltip only — tooltips on touch are problematic, and screen readers may not get the explanation. *Fix:* "Every rendered page MUST set `<html lang>` to the active locale. Mixed-locale fallback content must use inline `<span lang="sr">…</span>` wrappers so AT pronounces correctly. The ⓘ marker uses `<button aria-label="Sadržaj na srpskom — još nije preveden">` (focusable, screen-readable, not tooltip-only)."

- **Form error association incomplete** (EXPERIENCE.md §Forme, §Accessibility Floor). `aria-required`, `aria-invalid`, `aria-describedby` listed (good), but `aria-describedby` must link helper text AND error text by `id`. The "validation na blur" timing means errors appear after focus left field — SR users navigating with `Tab` may not hear the error unless live region is in place. Also: "Required polja označena vizuelno (*)" — the `*` needs `aria-hidden` so SR doesn't read "asterisk" (since `aria-required` covers it). *Fix:* Explicit pattern: `<input aria-required="true" aria-invalid="true" aria-describedby="field-help field-error">`. Asterisk wrapped `aria-hidden="true"`. Error region has `role="alert"` (or page-level summary list with skip-to-field links). Document focus-management on submit-with-errors (focus moves to first error field or summary).

- **Lime-500 must be banned as text on light surfaces** (DESIGN.md §Colors, §Don'ts already touches this). Measured: **lime-500 on white = 1.64:1, on cream = 1.45:1** — fails even large-text 3:1. Spec says "lime-500: hover na linkovima, highlights u akordionu". Hover-state link colors must still pass contrast (WCAG SC 1.4.3 applies to hover state of text). On `green-800`/`green-900` lime passes (6.9–7.1:1), but the accordion is on cream/white in product pages. *Fix:* Make explicit in DESIGN.md §Don'ts: "lime-500 MUST NOT be used as text color on any surface lighter than green-600. As a hover color for links on dark backgrounds only. For links on light surfaces, use green-800 underline pattern."

- **Count-up + sticky nav shrink + slider auto-advance + card hover lift = simultaneous-motion overload** (EXPERIENCE.md §Animacije). Reduced-motion is named but only for animation list; it does not say "*all* parallel motion sources stop". On a homepage, a user with vestibular sensitivity who hasn't toggled `prefers-reduced-motion` (e.g., a temporary state, low-end Android default) still gets 4+ concurrent motion sources. *Fix:* Add a "Motion budget" rule to §Animacije: no more than 2 concurrent decorative motion sources at a time per viewport. Auto-advance and count-up suspend when off-screen. Card hover lift uses transform only (already correct), preserving GPU compositing. Provide an in-page "reduce motion" toggle for users who can't set OS-level preference.

## Medium (10)

- **Repeating Element corner motif role is undefined for AT** (DESIGN.md §Repeating Element). "Tanke bele koncentrične četvrt-kružnice" is decorative per the Glossary metaphor, but the spec doesn't explicitly mark it `aria-hidden`. If implemented as inline SVG with a `<title>` it will be read out. *Fix:* Specify "Repeating Element corner motif: `aria-hidden="true"`, no `<title>`, role decorative. Brand-specific color (jeegee-blue) does not affect this; the shape is never an interactive target."

- **Jeegee-blue (#00A4E9) contrast on white is 2.80:1** (DESIGN.md §Brand-specific). Only used on Repeating Element hero on Jeegee pages — but if any text sits on that blue background (the hero overlay card or watermark text), it fails AA normal-text contrast for white text. *Fix:* Specify that text on Jeegee-blue Repeating Element backgrounds must use green-900 (passes only if dark variant exists — would need testing) or that the blue is *only* a decorative fill with no overlaid text. Add `aria-hidden` on the SVG.

- **Success color (#3a8a3a) on white = 4.31:1 fails AA normal-text** (DESIGN.md §Semantic mapping). If used as text color for success confirmation (e.g., "Hvala! Vaš upit je primljen." with success-color heading or check icon next to body text), it fails. *Fix:* Use #2d6e2d or darker for success text; keep #3a8a3a only as an icon fill ≥24px (which qualifies as non-text and passes 3:1).

- **Gray-300 border on white = 1.53:1** (DESIGN.md §Form input). Form input border uses gray-300 on white — fails 3:1 non-text contrast (WCAG 1.4.11). Empty/unfilled state of an input must be perceivable. *Fix:* Replace input border default with gray-700 or green-600 (≥3:1). Reserve gray-300 for non-essential dividers only.

- **Hero overlay card is text-on-image (potential contrast killer)** (DESIGN.md §Card hero-overlay, EXPERIENCE.md §Početna sekcija 1). Spec says "preko foto-pozadine traktora" with white text on green-800 *card*; the card is solid (OK). But if any hero text spills onto the photo, it fails. Also: `hero overlay card` is below an h1 image that may have visual noise behind the card — image needs treatment. *Fix:* Add guidance: "Hero photo behind overlay card must have a 30-40% black gradient under the card region to ensure card edges don't fade into busy photos. No text outside the solid green-800 card."

- **Skip link declared but no target/style guidance** (EXPERIENCE.md §Accessibility Floor). "Skip link „Preskoči na sadržaj" na vrhu strane (vidljiv pri Tab focusu)" — but spec doesn't define the target `#main` landmark, focus behavior on click, or visible style. *Fix:* Specify: skip link points to `<main id="main" tabindex="-1">`, becomes visible at 16px from top-left with green-800 background white text on focus, slides in 100ms. Test it as the first Tab stop.

- **Wave divider SVG accessibility unspecified** (DESIGN.md §Wave divider). Likely decorative but spec is silent. *Fix:* "All decorative SVG dividers: `aria-hidden="true"`, no `<title>`. Mark `role="presentation"` if rendered as inline element."

- **Empty state pattern lacks heading semantics** (EXPERIENCE.md §State Patterns, §Empty states). "Ikona + naslov + opis + CTA" — naslov should be a heading (h2 or h3) for AT users navigating by headings. Also: empty state must not visually replace results without an announcement. *Fix:* "Empty state markup: `<section role="status"><h2>{poruka}</h2><p>…</p><button>CTA</button></section>`. When filter swap leaves zero results, announce via `aria-live`."

- **Touch target sizing not enforced for icon-only controls** (EXPERIENCE.md §Click vs Tap targets). 44×44px stated, but: language switcher (SR/HU/EN), social icons, search-icon-then-input, hamburger, prev/next slider arrows, accordion +/− — these are icon-only controls and likely fail 44px without explicit padding. Top-header phone icon (mobile) — single-tap-to-expand could be tiny. *Fix:* Explicit table mapping every icon-only control to its minimum touch target. Add: "Icon-only buttons: 44px hit area achieved through padding even if visual icon is 16-24px."

- **HTMX loading state visually opaque (0.5) — but no `aria-busy`** (EXPERIENCE.md §Filter, §State Patterns). Filter area gets opacity 0.5; SR users get no signal that content is updating. *Fix:* "Region under HTMX request gets `aria-busy="true"` until swap completes. Coupled with `aria-live='polite'` on result container so result count is announced after swap."

## Low (7)

- **`prefers-reduced-motion` doesn't address auto-advance carousel pause** (EXPERIENCE.md §Interaction Primitives → Reduced motion). Lists count-up, smooth scroll, accordion — but not "auto-advance pauses entirely". *Fix:* Add slider auto-advance to the reduced-motion list explicitly.

- **Card "klik se proteže celom karticom" risks nested-interactive markup** (EXPERIENCE.md §Card). If entire card is a link AND a "CTA button" sits inside, it nests interactive elements — invalid HTML, AT trip. *Fix:* Use a single `<a>` wrapping the whole card with the CTA visualized but not a separate `<button>`. Or: use card-area click via JS but mark the CTA as the *only* focusable target inside.

- **Footer newsletter input (if added) and social icons need labels** (EXPERIENCE.md §Footer). Stated structure shows social icons; labels not specified. *Fix:* "Social icon links: `aria-label="Ćorić Agrar na Facebook-u"` etc. Newsletter input (if/when added): visible label, no placeholder-only."

- **Required indicator `*` could conflict with autofill UI** (EXPERIENCE.md §Accessibility Floor → Forme). Asterisk before/after label — Vi-form Serbian users might also need a legend like "* označava obavezna polja" once at top of form. *Fix:* Add legend pattern: `<p id="form-required-note">Polja označena * su obavezna.</p>` referenced by form `aria-describedby`.

- **Auto-popunjeno Model field UX risk for SR users** (Flow 1, step 5). "Polje *Model* je auto-popunjeno" — AT user may not realize it. *Fix:* When auto-filling from URL context, announce: `aria-live="polite"` text: "Model je automatski popunjen na osnovu trenutnog proizvoda."

- **Print stylesheet deferred but accessibility implication noted** — print users often have low vision (large monospace, high contrast). *Fix:* Defer acknowledged; flag for v2 a11y review.

- **Internationalization edge cases noted but `bdi`/`dir` not mentioned** — Hungarian and English don't need RTL, but mixed-locale brand names (e.g., "Saillong by Maki") in Cyrillic Serbian text may need `<span lang="en">`. Also: hyphenation differs by language. *Fix:* Add to §Accessibility Floor: "Mixed-locale text inline-wrapped with `<span lang>`. Hyphenation via CSS `hyphens: auto` only after `lang` set on container."

## Bonus: Inclusive design beyond WCAG

- **Cognitive load on Početna is high**: hero overlay + sticky nav transformation + count-up statistics + auto-advance testimonials + wave dividers + Repeating Element watermarks. For users with ADHD, anxiety, or cognitive disabilities (not all of whom set `prefers-reduced-motion`), this is a lot. Consider an "Easy mode" toggle (persistent localStorage) that disables non-essential motion and stickiness.

- **Agricultural-sector users skew older** (40-65+, per typical persona for tractor buyers). Body size is 20px (above default 16px) — excellent. But the hero h1 at 56px with Roboto Light (300) is *thin* — light weights at large sizes can be hard for older eyes, especially against textured photos. Consider Roboto Regular (400) for hero on cream/light backgrounds, reserving 300 for green-800 surfaces where the contrast does the work.

- **Form Vi-form vs Ti-form switch** (EXPERIENCE.md §Lokalizacija tone) is good cultural localization but creates a register inconsistency for cognitively-fatigued users. Document this explicitly to AT translators and content authors.

- **Mobile servicing flow (Flow 2)** is on a tractor in the field — likely glove use, sunlight glare, intermittent connectivity. Implications: larger touch targets than 44px (consider 56px), high-contrast outdoor mode, offline-graceful form (save-draft-locally before submit, queue if offline). Not WCAG but real-world inclusive design.

- **Language switcher SR/HU/EN uses 2-letter abbreviations** — for non-native speakers these are codes. Consider full language names on hover/focus: "Srpski / Magyar / English" exposed via `aria-label` or visible toggle.

- **Auto-advance testimonials are a poor pattern even with pause** — testimonials are read content, and rotating them mid-read is hostile. Consider a static stack with "Vidi sve" link, or only advance on user click. Honors WCAG and respects reading flow.

- **PDF brochure download accessibility** — the brochure PDFs themselves must be tagged and screen-reader accessible. Spec doesn't mention this, but a "Preuzmi brošuru" download to an inaccessible PDF defeats the purpose. *Suggestion:* Add to content guidelines: "All downloadable PDFs MUST be tagged PDF/UA-conformant. Untagged brochures should be replaced or supplemented with an HTML specs page."

- **Error message Serbian-first** is correct, but if user has system in English and switches to EN locale, error text in the validation message ("Email format nije ispravan") must localize. Note for content audit.
